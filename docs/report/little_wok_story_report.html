ABSTRACT

The rapid adoption of online ordering has transformed the food service industry, especially for cloud kitchens that depend entirely on digital workflows for customer acquisition, order processing, and delivery coordination. This project presents Little Wok Story, a production-oriented cloud kitchen ordering platform designed for Indo-Chinese and Pan-Asian cuisine. The platform supports customer ordering, menu browsing, cart and checkout, order history, role-based administration, and payment integration.

The system has been developed using a modern web architecture with a Next.js 14 frontend and a FastAPI backend. The frontend delivers a responsive and branded user experience, while the backend provides secure REST APIs for authentication, menu management, order creation, coupon validation, and payment workflows. A normalized PostgreSQL schema has been designed to manage users, addresses, menu items, coupons, orders, and order items. Authentication is implemented using JSON Web Tokens (JWT), and security considerations such as password hashing, input validation, role-based access control, and rate limiting have been incorporated.

The project addresses both customer-side and business-side requirements. Customers can search dishes, customize orders, maintain saved addresses, and place orders through online payments or cash on delivery. Administrators can manage menu availability, monitor order flow, review user activity, and track business performance through a protected dashboard. The proposed system demonstrates how cloud kitchen operations can be digitized using a scalable full-stack architecture suitable for future extension with real-time order tracking, notifications, multi-location support, and advanced analytics.

CHAPTER 1

INTRODUCTION

Food ordering has shifted from a traditional in-person activity to a predominantly digital experience. Cloud kitchens, unlike conventional restaurants, rely almost entirely on online channels for visibility, order management, and revenue. This change creates a strong need for reliable software platforms that connect customers, kitchen operations, and business administration within a single system.

Little Wok Story is conceived as a dedicated food ordering platform for a cloud kitchen specializing in Indo-Chinese and Pan-Asian cuisine. The project focuses on building a usable, scalable, and secure solution that supports customer ordering, menu exploration, checkout, order tracking, and business-side administration. The objective is not only to create a visually appealing website, but also to implement the core backend logic and data design needed for real-world deployment.

The system has been designed around three primary user roles: customers, administrators, and kitchen staff. Customers require a smooth browsing and checkout experience. Administrators require control over menu items, orders, coupons, and analytics. Kitchen staff require visibility into incoming orders and preparation status. These needs make the project a practical case study in full-stack application development.

1.1 Problem Statement

Many small and medium cloud kitchens depend on third-party platforms that charge high commissions and offer limited brand control. They often lack custom workflows, direct customer ownership, and role-specific management tools. This project aims to provide an independent ordering platform that allows a cloud kitchen to manage branding, ordering, operations, and customer engagement more effectively.

1.2 Objectives of the Project

  1. To design and implement a responsive food ordering website for a cloud kitchen brand.
  2. To build secure backend APIs for authentication, menu management, order creation, and payments.
  3. To design a normalized relational database schema for operational consistency and scalability.
  4. To provide separate access flows for customers, administrators, and kitchen staff.
  5. To create a system architecture that can be extended in production with notifications, analytics, and real-time updates.

1.3 Scope of the Project

The current implementation covers customer ordering, cart management, checkout flow, order history, admin panel scaffolding, kitchen display view, coupon support, and payment integration endpoints for Razorpay and Stripe. The scope excludes full production deployment, real SMS/WhatsApp notifications, and real-time live tracking, though the architecture supports their addition in future iterations.

CHAPTER 2

LITERATURE STUDY

The development of online food ordering systems has been influenced by research in e-commerce, human-computer interaction, distributed systems, and mobile commerce. Existing platforms such as Swiggy, Zomato, Uber Eats, and restaurant-owned systems illustrate how digital ordering must balance speed, usability, transaction safety, and operational coordination.

Prior software engineering studies emphasize the value of modular design, layered architecture, and role-based system access in transactional applications. In food ordering domains, common challenges include maintaining menu accuracy, reducing checkout friction, ensuring payment reliability, and supporting operations across customer and staff interfaces. A well-structured architecture therefore needs clear separation between presentation, application logic, and persistence layers.

Modern web stacks increasingly combine a React-based frontend with lightweight API services and relational databases. Next.js is widely adopted for fast, SEO-friendly interfaces, while FastAPI has emerged as a strong backend framework for Python due to its performance, clean routing model, and type-driven validation. SQL databases remain suitable for order-centric systems because food ordering workflows are strongly transactional and relational by nature.

2.1 Study of Existing Systems

  • Aggregator platforms provide reach but reduce direct brand ownership and profitability through commissions.
  • Simple static restaurant websites provide branding but often lack operational workflows such as status management and coupon logic.
  • Custom admin systems improve control but require careful attention to security, validation, and scalability.

2.2 Motivation for the Proposed System

The proposed system combines branded customer experience with direct control over menu, pricing, orders, and business logic. It is especially suitable for a cloud kitchen that wants independence from third-party marketplaces while maintaining room for operational growth.

CHAPTER 4

METHODOLOGY AND IMPLEMENTATION

The project follows a modular development methodology in which the frontend, backend, and database layers are designed as separate but coordinated components. This approach improves maintainability and allows independent scaling of major system parts. The implementation was guided by the principle of clear responsibility: the frontend manages user interaction, the backend manages rules and validation, and the database persists relational state.

4.1 Frontend Implementation

The frontend is built using Next.js 14 with the App Router. Core user-facing pages include the landing page, menu page, cart, checkout, dashboard, admin panel, and kitchen display. Tailwind CSS is used for styling, while Framer Motion supports subtle interface animations. Zustand is used to manage cart state in the browser, including quantity changes and coupon codes.

4.2 Backend Implementation

The backend is developed using FastAPI. REST endpoints are grouped by domain, such as authentication, menu, addresses, coupons, orders, payments, and administration. Pydantic schemas validate request and response data, while SQLAlchemy models map application entities to relational tables. Dependencies are used to enforce authenticated and role-protected access.

4.3 Security Implementation

  • Passwords are hashed using bcrypt before storage.
  • JWT tokens are used to authenticate protected requests.
  • Rate limiting is applied to sensitive endpoints such as login and order creation.
  • Role-based dependencies protect admin and kitchen routes.
  • CSRF checks and validation layers reduce common web risks.

4.4 Payment Workflow

The payment layer supports Razorpay for Indian transactions and Stripe as a global fallback. The backend exposes endpoints to create payment sessions and verify completed payments. Verification is handled on the server side to ensure that order status is not updated purely from frontend signals. A COD path is also supported for operational flexibility.

4.5 Order Processing Workflow

  1. User authenticates and stores a valid session token.
  2. User selects food items, customizations, and quantities.
  3. User submits delivery address and checkout request.
  4. Backend validates menu availability, coupon eligibility, and totals.
  5. Order and order items are stored in the database.
  6. Payment is initiated or COD is selected.
  7. Admin and kitchen flows can update the order through preparation and delivery states.

The developed system successfully demonstrates the key workflows expected from a branded cloud kitchen ordering platform. The backend supports authentication, order processing, and protected administrative operations.

5.2 Observations

The separation of concerns between frontend and backend makes the system easier to extend and debug. The use of a relational database is particularly effective because order workflows depend on strong relationships between users, menu items, and transaction records. The chosen architecture also supports future migration to production hosting platforms without major redesign.

5.3 Limitations

  • Real-time socket-based order updates are not yet enabled.
  • Production email, SMS, or WhatsApp notification channels require live credentials.
  • Image upload and storage are currently represented as URL-based content rather than a full media pipeline.
  • Advanced analytics and loyalty programs remain future enhancements.

CHAPTER 6

CONCLUSION AND FUTURE WORK

This project demonstrates the design and development of a full-stack food ordering platform for a cloud kitchen brand. The system combines customer-facing usability with backend reliability and administrative control. It shows how a modern web stack can be used to build an independent ordering solution that reduces dependence on third-party aggregators and strengthens direct brand ownership.

From a software engineering perspective, the project highlights the practical value of modular architecture, typed interfaces, secure API design, and normalized data modeling. It also illustrates how role-based access can support multiple stakeholders within the same digital platform, including customers, admins, and kitchen operators.

Future work may include real-time order notifications, live kitchen dashboards, loyalty points, AI-based recommendations, push notifications, multi-location support, and deployment to managed cloud platforms. With these extensions, the platform can evolve from an academic prototype into a market-ready product for operational use.
